## 2. 주요 기술 결정와 이유

### 2.1 이벤트 기반 아키텍처 (Event Bus)

| 대안 | 동기 직접 호출 | → | 이벤트 버스 |
|------|------------|------|-----------|
| 결합도 | 강한 결합 | | 묶음 |
| 도메인 경계 | 흐름 보기 힘들음 | | 선명함 |
| 추가/삭제 | 기존 코드 수정 필요 | | 발행/구독만 추가 |
| 유닛테스트 | 어려움 | | 단위 테스트 매우 쉽게 |

**배우기**: 도메인 간 갯깒히 연결되어 있으면 이벤트 버스 고려

### 2.2 O(1) 캐시 (dict)

| 지표 | 목표 | 실제 | 비교 |
|------|------|------|------|
| 조회 속도 | O(1) | O(1) | dict = hash table |
| 메모리 | 적게 | ~78KB/10만 건 | 데이터베이스 조회보다 1000x 빠름 |
| 구현 난이도 | 쉽음 | 쉽음 | Python 내장 dict 활용 |

**배우기**: 게임 서버의 가장 큰 보트릭 = 자원 조회. 여기서 쾌나면 루키로쟁.

### 2.3 비동기 처리 (asyncio)

| 경우 | 동기 | → | 비동기 |
|------|------|------|--------|
| 100명 동시 접속 | 하나씩 처리, 데드락 | | 모든 접속을 동시에 관리 |
| 탐색 요청 | 기다려야 함 | | 보내놓고 다른 일 하다가 결과 받음 |
| CPU 사용률 | 쉽게 100% 돼버림 | | I/O 대기 중 CPU 사용 안 함 |

## 3. 최적화 원칙

### 메모리 vs 속도 트레이드오프
이 시스템은 메모리를 더 쓰고 속도를 사셨음.

**이유**: 방수 게임 서버에서 베이스 제약은 배튼이 아닌 I/O 입니다.
메모리 1GB 씀 = 최소 밀만 명의 자원 데이터를 O(1)에 저장가능.
속도 가성비 = 보다 낙은 유저 경험. 여기서 메모리를 더 쓰는 건 합릿값.
